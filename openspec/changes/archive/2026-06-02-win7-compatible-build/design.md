## Context

当前项目使用Python 3.11 + PyInstaller打包，生成的exe文件在Windows 11上运行正常，但在Windows 7上无法运行。错误原因是Python 3.11依赖的`api-ms-win-core-path-l1-1-0.dll`在Windows 7中不存在。

项目设计要求兼容Windows 7及更高版本，但实际打包环境使用了不兼容的Python版本。需要建立一个自动化打包流程，确保为不同Windows版本提供兼容的可执行文件。

## Goals / Non-Goals

**Goals:**
- 建立GitHub Actions自动化打包流程
- 使用Python 3.7环境打包Windows 7兼容版本
- 保持现有Windows 10/11版本的打包流程
- 自动发布到GitHub Releases
- 用户无需手动安装补丁即可运行

**Non-Goals:**
- 修改现有代码以兼容Python 3.7（代码已兼容）
- 支持Windows XP或更早版本
- 创建安装程序或自动更新功能
- 修改现有打包配置（保持向后兼容）

## Decisions

### 决策1：使用Python 3.7打包Windows 7版本
**选择**: 使用Python 3.7环境打包Windows 7兼容版本
**理由**: Python 3.7是最后完全支持Windows 7的版本，不依赖`api-ms-win-core-path-l1-1-0.dll`
**替代方案**: 
- 方案A: 在Windows 7上安装KB2999226补丁（需要用户手动操作）
- 方案B: 手动放置DLL文件（不可靠，用户可能找不到正确版本）

### 决策2：使用GitHub Actions进行自动化打包
**选择**: 使用GitHub Actions作为CI/CD平台
**理由**: 
- 免费且集成于GitHub
- 支持Python 3.7环境
- 可自动创建Release
- 社区支持良好
**替代方案**: 
- 方案A: 本地打包脚本（需要开发者手动执行）
- 方案B: 其他CI/CD平台（如GitLab CI、Jenkins）

### 决策3：同时发布多个Windows版本
**选择**: 为Windows 7和Windows 10/11分别打包
**理由**: 
- 最大化兼容性
- 用户可根据系统选择合适版本
- 避免"一刀切"导致的兼容性问题
**替代方案**: 
- 方案A: 只打包Windows 7版本（可能影响现代系统性能）
- 方案B: 只打包Windows 10/11版本（放弃Windows 7用户）

### 决策4：使用现有的PyInstaller配置
**选择**: 基于现有的`发票批量重命名工具_v2.spec`配置
**理由**: 
- 配置已经过测试，功能完整
- 只需修改Python版本，无需重新配置
- 保持一致性
**替代方案**: 
- 方案A: 创建全新的配置文件（增加维护成本）
- 方案B: 使用命令行参数打包（灵活性较低）

## Risks / Trade-offs

### 风险1: Python 3.7依赖兼容性
**风险**: 某些依赖库可能不支持Python 3.7
**缓解措施**: 
- 测试所有依赖在Python 3.7上的兼容性
- 使用虚拟环境隔离，确保依赖版本正确
- 准备回退方案，如果发现不兼容依赖

### 风险2: GitHub Actions运行时间限制
**风险**: 免费账户有运行时间限制
**缓解措施**: 
- 优化打包流程，减少不必要的步骤
- 使用缓存加速依赖安装
- 只在必要时触发（如推送标签时）

### 风险3: 用户混淆版本
**风险**: 用户可能下载错误版本
**缓解措施**: 
- 在Release页面明确标注版本兼容性
- 文件名包含版本信息（如`_Win7`、`_Win10`）
- 更新README说明不同版本的使用场景

### 风险4: 维护成本增加
**风险**: 需要维护两个打包流程
**缓解措施**: 
- 使用矩阵策略，减少重复配置
- 自动化测试确保两个版本功能一致
- 文档化打包流程，便于维护

## Migration Plan

如果出现问题：
1. 禁用GitHub Actions工作流
2. 恢复到手动打包流程
3. 修复问题后重新启用
